# Звіт щодо контейнеризації ML-моделі

## Порівняння Docker-образів

| Характеристика      | `ml-fat`                                                       | `ml-slim`                                                 |
| :------------------ | :------------------------------------------------------------- | :-------------------------------------------------------- |
| **Базовий образ**   | `python:3.9`                                                   | `python:3.9-slim` (Multi-stage)                           |
| **Розмір**          | 2.69 GB                                                        | 1.01 GB                                                   |
| **Кількість шарів** | Більше (через важкий базовий образ та відсутність оптимізації) | Менше (базовий образ slim, використання multistage build) |

### Проблеми "Fat" образу:

1. **Зайві інструменти:** Базовий образ `python:3.9` містить багато системних утиліт (C-компілятори, curl, git), які не потрібні для простого інференсу. Це збільшує attack surface (вразливості).
2. **Розмір PyTorch:** За замовчуванням `pip install torch` тягне бінарники з підтримкою CUDA (GPU), що додає понад 1-2 ГБ зайвої ваги, навіть якщо сервер працює лише на CPU.
3. **Кеші:** Не очищені кеші `pip` додають зайві мегабайти у шари.

### Пропозиції з подальшої оптимізації:

1. **Використання Distroless:** Перехід на Google Distroless образи (наприклад, `gcr.io/distroless/python3`) залишить лише саму програму та Python runtime, без доступу до shell (bash/sh), що робить контейнер максимально безпечним.
2. **TorchScript-lite або ONNX:** Конвертація моделі у формат ONNX Runtime або використання PyTorch Mobile (lite) дозволить повністю відмовитись від "важкої" бібліотеки `torch` в runtime.
3. **Очищення кешу apt/pip:** Використання прапорця `--no-cache-dir` в pip та `rm -rf /var/lib/apt/lists/*`.
